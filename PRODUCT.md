# Product

## Register

product

## Users

Solo developers and founders who run customer support themselves. They work at a desk, often alone, switching between supervising AI agents and stepping in on tickets directly. They need to see what agents have done, what still needs human judgment, and move fast without a support team around them.

## Product Purpose

Ticqex is agent-first support infrastructure: a realtime Kanban admin where humans supervise, while agents operate tickets through a typed REST API and MCP server. Success means the admin feels like a serious control surface for that split workflow — agent activity is visible and legible, human intervention is one click away, and the UI never pretends agents are an afterthought.

## Brand Personality

Technical · Direct · Quiet

Voice is utilitarian and confident, not salesy. Copy states what happened and what to do next. Emotional goal: calm competence — the feeling that the system is working and you can trust what you see.

Reference feel: Linear — tight hierarchy, fast scanning, opinionated density without clutter.

## Anti-references

- Generic SaaS helpdesk clones (Intercom, Zendesk patterns): purple gradients, hero metrics, identical feature-card grids, marketing filler inside admin surfaces.
- AI-slop scaffolding: warm cream body backgrounds, gradient text, small-caps eyebrows on every section, numbered section markers used as decoration.
- Consumer/playful UI: mascot energy, bubbly rounded everything, loud accent colors, delight-for-delight's-sake motion.

## Design Principles

1. **Agents are first-class operators** — Actor, channel, and automation state should be scannable on cards and in detail views; the UI reflects parity between agent and human actions, not "AI as a badge."
2. **Quiet competence** — Density serves speed for a solo operator; no subtitles that restate headings, no decorative chrome that competes with ticket data.
3. **Show system state** — Status, assignment, last activity, and source channel answer "what's happening?" at a glance before opening a ticket.
4. **Practice what you preach** — The admin should feel like the same infrastructure the API and MCP expose: precise, dependable, no toy energy.
5. **Direct paths** — Every screen has an obvious next action (triage, reply, reassign, configure); avoid ceremony between intent and execution.

## Accessibility & Inclusion

WCAG 2.1 AA for the admin UI: sufficient contrast on body and placeholder text, full keyboard operability for board and modals, visible focus states, and `prefers-reduced-motion` respected for all animations.
